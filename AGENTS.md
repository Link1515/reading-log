# AGENTS.md

## 專案概要

這是一個「個人閱讀紀錄網站」，用來整理與展示我所有已讀書籍。

本專案是純前端靜態網站：

- 不使用後端
- 不使用資料庫
- 不依賴 server-side API
- 所有資料都儲存在靜態檔案中
- 使用 Vue 開發

代理在進行任何實作時，必須優先維持「可直接部署為靜態網站」這個前提。

---

## 產品目標

網站需要讓使用者可以：

- 查看所有讀過的書
- 依書名搜尋
- 依作者搜尋
- 依閱讀日期做區間查詢
- 對結果做排序

網站的整體體驗應偏向：

- 深色書店／藝文空間風
- 安靜、內斂、成熟
- 重視排版、留白、閱讀感
- 不要有強烈 AI 產品感
- 不要做成 dashboard 或科技感 SaaS 首頁

重點是簡單、清楚、穩定、容易維護，同時保有書店、文化空間、閱讀檔案館的氣質。

## 視覺風格要求

### 整體風格定位

本專案的 UI 風格必須採用：

- 深色書店／藝文空間風
- 深色編輯感
- 文化內容導向
- 低干擾、高可讀性

畫面應讓人聯想到：

- 獨立書店
- 藝文展覽網站
- 文化刊物
- 深色閱讀檔案館

而不是：

- AI 工具站
- 科技產品官網
- 管理後台
- 霓虹感 dashboard

### 色彩原則

整體必須使用深色模式，但避免過於冰冷或科技化。

建議色彩方向：

- 背景：炭黑、墨黑、深灰黑、帶一點棕調或墨綠調的深色
- 主文字：米白、暖灰白、柔和淺灰
- 次文字：中灰、偏暖灰
- 點綴色：低飽和暗金、銅色、霧藍、灰綠，僅少量使用

避免：

- 霓虹藍
- 紫藍漸層發光
- 過亮青色
- 高飽和科技色
- 大片玻璃擬態效果

### 排版原則

本專案必須重視 typography：

- 書名應是主要視覺焦點
- 作者、類別、日期為次層資訊
- 以字級、字重、留白建立層次
- 不依賴過多裝飾元素建立美感
- 整體要像編輯設計或藏書目錄，而不是卡片式 SaaS 介面

### 元件視覺原則

- 控制元件要低調、克制
- 邊框可有，但要細且不刺眼
- hover、focus 狀態要清楚但不誇張
- 陰影應盡量少用或非常淡
- 避免發光效果
- 避免玻璃感、漂浮卡片感、強烈景深效果

### 動效原則

- 動效必須極少
- 只允許非常輕微的淡入、顏色變化、背景變化
- 不要使用誇張位移、彈跳、霓虹特效
- 不要讓動效影響閱讀節奏

### 視覺關鍵字

代理在設計與實作畫面時，應參考以下關鍵字：

- dark editorial
- bookstore
- archive
- arts and culture
- typography-first
- quiet interface
- warm dark theme

## 資料來源與限制

### 資料保存方式

所有書籍資料必須保存在靜態檔案中，不可改成後端資料來源。

本專案區分兩種資料檔：

- 原始資料：供人手動維護
- 產出資料：供前端程式直接使用

建議使用：

- `src/data/raw/books.json`：原始資料
- `src/data/generated/books.json`：build 流程整理後產出的資料

原始資料檔是唯一人工維護來源，前端不直接依賴原始資料檔，而是依賴 build 後產生的資料檔。

### 書籍資料欄位

每筆書籍資料至少包含：

- `title`: 書名
- `author`: 作者
- `category`: 類別
- `readDate`: 閱讀日期，格式固定為 `YYYY-MM-DD`

範例：

```json
[
  {
    "title": "原子習慣",
    "author": "James Clear",
    "category": "自我成長",
    "readDate": "2025-01-15"
  },
  {
    "title": "人類大歷史",
    "author": "Yuval Noah Harari",
    "category": "歷史",
    "readDate": "2025-03-02"
  }
]
```

### 日期要求

- 日期格式統一使用 ISO 格式：`YYYY-MM-DD`
- 日期區間比較必須可靠，不可只做鬆散字串處理
- 若遇到空值或格式不正確資料，應避免讓畫面崩潰

## 必要功能

### 1. 書籍列表

必須有主列表頁面顯示所有書籍，至少展示：

- 書名
- 作者
- 類別
- 閱讀日期

### 2. 書名搜尋

必須支援用書名關鍵字搜尋：

- 預設採部分比對
- 不要求完全相符
- 建議不分大小寫
- 搜尋結果應即時更新

### 3. 作者搜尋

必須支援用作者名稱搜尋：

- 預設採部分比對
- 不要求完全相符
- 建議不分大小寫
- 搜尋結果應即時更新

### 4. 日期區間查詢

必須支援依閱讀日期做區間篩選：

- 起始日期
- 結束日期

規則：

- 只設定起始日期時，顯示該日之後的書
- 只設定結束日期時，顯示該日之前的書
- 同時設定時，顯示區間內的書
- 區間為包含邊界

### 5. 排序

必須支援至少以下排序方式：

- 閱讀日期新到舊
- 閱讀日期舊到新
- 書名 A 到 Z
- 書名 Z 到 A
- 作者 A 到 Z
- 作者 Z 到 A

### 6. 條件可組合

搜尋、日期區間、排序必須可以同時生效。

例如：

- 先搜尋書名或作者
- 再篩日期區間
- 最後排序結果

## UI / UX 要求

### 介面原則

- 簡潔
- 易讀
- 操作直覺
- 手機與桌機都可正常使用
- 預設採深色介面
- 視覺風格參考 Fluent UI 深色模式

### 視覺風格基準

後續代理在調整或擴充前端畫面時，應優先延續目前已確立的深色設計語言，避免每次 session 重新切換風格。

具體原則如下：

- 整體使用深色背景，不回退成亮色主題
- 視覺參考 Fluent UI 深色模式的層次與對比
- 卡片、篩選區、空狀態等區塊使用深色半透明 surface
- 文字以高可讀性的淺色為主，次要資訊使用較低對比但仍清楚可讀的灰階色
- 強調色以 Fluent 風格的藍色系為主，不使用高飽和雜色混搭
- 輸入框、按鈕、卡片邊框需有清楚但不厚重的邊界層次
- focus 狀態要清楚，方便鍵盤操作與辨識
- 保持簡潔、穩定、偏系統化的視覺，不追求過度裝飾
- 若新增元件，需延續既有圓角、陰影、邊框與色彩層級
- 重視內容優先
- 工具列存在感低，但操作明確

### 畫面配置建議

在書籍列表上方提供控制區，至少包含：

- 一個全域搜尋入口
- 起始日期輸入
- 結束日期輸入
- 排序選單
- 清除篩選按鈕

搜尋入口的互動基準如下：

- 預設使用單一搜尋框，不再拆成「書名搜尋」與「作者搜尋」兩個欄位
- 搜尋框查詢目標同時包含書名與作者
- 搜尋框預設可隱藏，作為一個獨立的搜尋浮層
- 桌機版支援按 `/` 開啟搜尋、按 `Esc` 關閉搜尋
- 手機版不顯示鍵盤快捷鍵提示文字
- 手機版與桌機版都保留右下角 fixed 搜尋按鈕，作為開啟搜尋的明確入口
- 桌機版可在搜尋按鈕 hover 時用 tooltip 提示 `/` 與 `Esc` 快捷鍵
- 日期區間、排序、清除條件應放在搜尋框下方，並以 collapse 形式開闔
- 搜尋框展開時，背景應有低調的毛玻璃遮罩效果，方向可參考 Fluent UI 深色浮層

控制區應：

- 視覺上低調
- 不喧賓奪主
- 與列表內容保持足夠留白
- 看起來像書店檢索介面，而不是 SaaS 篩選面板

搜尋浮層與條件區的動效基準如下：

- 搜尋浮層出現時使用淡入，並帶有輕微由下往上的進場感
- 搜尋浮層隱藏時使用淡出，不使用誇張位移
- 條件區的 collapse 動畫必須輕量、短促、不影響閱讀節奏

### 列表呈現建議

書籍列表優先採用以下方向：

- 垂直列表
- 清楚的排版節奏
- 書名最大
- 作者與日期為輔助資訊
- 類別可用低調標籤或文字顯示

不建議預設使用過度厚重的卡片牆式設計。若使用卡片，也必須保持非常克制。

### 狀態呈現

介面應清楚反映目前狀態：

- 關鍵字搜尋內容
- 日期區間
- 排序方式
- 篩選後結果數量

### 空狀態

當沒有符合條件的書籍時，顯示明確提示，例如：

- `找不到符合條件的書籍`

空狀態畫面必須保持克制與整體風格一致，不可出現過度插畫化或產品化的空畫面。

## 技術要求

### 框架

- 使用 Vue
- 優先採用簡潔且現代的 Vue 寫法
- 可使用 Composition API

### 狀態與資料處理

應將「原始資料」與「畫面衍生結果」分開處理：

- 原始資料：build 前的人工維護資料

目前搜尋實作基準：

- 使用 `fuse.js`
- 單一關鍵字同時搜尋 `title` 與 `author`
- 搜尋結果即時更新
- 搜尋之後再套用日期區間與排序

### 元件拆分建議

可依需求拆成以下元件：

- `BookList`
- `BookListItem`
- `BookFilters`
- `SearchBar`
- `SortSelect`

不要求過度拆分；若專案規模小，應避免為拆而拆。


若使用不同結構，必須保持清楚、一致、易維護。

## 後續加入書籍資料的規劃

本專案必須以「之後由我直接修改 `.json` 檔案新增書籍，而不是透過 agent 或後台介面」為前提設計。

代理在實作時，必須讓整體流程符合以下原則：

- 我只需要維護原始資料檔
- 前端只讀取整理後的產出資料檔
- build 時自動完成資料整理、排序與基本驗證
- 原始資料即使順序混亂，build 後產物仍保持穩定一致

### 單一人工維護來源

未來新增、修改、刪除書籍資料時，唯一需要手動編輯的檔案是：

- `src/data/raw/books.json`

禁止要求我在新增書籍時同步修改：

- 前端元件
- 排序程式碼
- 搜尋程式碼
- 篩選程式碼
- 額外索引檔
- 類別對照表
- 其他手動同步檔案

### 人工維護原始資料的方式

未來每次新增一本書時，流程如下：

1. 打開 `src/data/raw/books.json`
2. 新增一筆書籍物件
3. 填入 `title`、`author`、`category`、`readDate`
4. 儲存檔案
5. 執行 build
6. build 自動整理出前端使用的 `src/data/generated/books.json`

### 原始資料檔的設計原則

原始資料檔是給人直接編輯的，因此必須：

- 可讀性高
- 欄位命名固定
- 不要求人工排序完全正確
- 容許之後資料量逐漸增加
- 不要求人工維護複雜索引或額外統計欄位

也就是說，`raw/books.json` 的目標是「好維護」，不是「前端最省成本」。

### 產出資料檔的設計原則

`generated/books.json` 是給前端使用的，因此 build 後應具備：

- 結構穩定
- 已排序
- 欄位完整
- 適合前端直接載入
- 不需要前端在執行時做大量整理

前端畫面應優先讀取：

- `src/data/generated/books.json`

不應直接讀取：

- `src/data/raw/books.json`

## Build 資料整理流程要求

代理必須規劃並實作一個 build 前置或 build 內整合流程，用來處理書籍資料。

建議建立：

- `scripts/build-books.js`

並在專案 build 時先執行這個資料整理腳本。

### Build 流程必須完成的事情

build 資料整理流程至少要做以下處理：

1. 讀取 `src/data/raw/books.json`
2. 驗證每筆資料欄位是否存在
3. 驗證 `readDate` 是否為合法 `YYYY-MM-DD`
4. 清理字串前後空白
5. 統一輸出欄位結構
6. 依固定規則排序
7. 產出 `src/data/generated/books.json`

### Build 時的排序規則

產出資料檔必須固定排序，避免原始資料順序混亂影響前端結果。

預設排序規則：

1. `readDate` 由新到舊
2. 若 `readDate` 相同，依 `author` 升冪
3. 若 `author` 相同，依 `title` 升冪

這個排序規則必須由 build 腳本自動套用，不依賴人工手動排序。

### Build 時的資料清理規則

build 腳本應盡量做輕量但穩定的資料整理，包括：

- 去除 `title`、`author`、`category` 前後空白
- 去除空字串欄位造成的異常
- 確保輸出格式一致
- 避免前端因原始資料小錯誤而直接崩潰

### Build 時的資料驗證規則

build 腳本至少要檢查以下條件：

- `title` 必須存在且為非空字串
- `author` 必須存在且為非空字串
- `category` 必須存在且為非空字串
- `readDate` 必須存在且符合 `YYYY-MM-DD`
- 原始資料格式必須是陣列

若資料不合法，build 應直接失敗並回報明確錯誤，讓我可以回去修正 `raw/books.json`。

### 重複資料檢查

build 腳本可加入輕量重複檢查。

預設判定為可能重複的條件：

- `title` 相同
- `author` 相同
- `readDate` 相同

若三者都相同，應視為疑似重複紀錄。

建議行為：

- 預設讓 build 失敗並提示重複資料位置
- 若專案後續明確需要，也可改成警告模式

### 前端資料格式要求

產出的 `generated/books.json` 至少要保留以下欄位：

- `title`
- `author`
- `category`
- `readDate`

若有額外為前端便利而加入欄位，例如：

- `id`
- `searchText`
- `readYear`

可以加入，但必須遵守：

- 原始資料仍維持簡單
- 額外欄位由 build 自動產生
- 不要求我手動維護這些欄位

### 建議自動產生欄位

若能提升前端使用便利性，可在 build 階段自動產生：

- `id`: 穩定且可重現的識別值
- `searchText`: 合併書名與作者的小寫搜尋字串
- `readYear`: 從 `readDate` 提取年份
- `readMonth`: 從 `readDate` 提取月份

但這些都是 build 衍生欄位，不可要求我手動填寫。

## 人工維護資料的規範

### 原始資料欄位格式

每次新增書籍時，欄位格式必須一致：

- `title`: 字串，不可為空
- `author`: 字串，不可為空
- `category`: 字串，不可為空
- `readDate`: `YYYY-MM-DD`

範例：

```json
{
  "title": "深度工作力",
  "author": "Cal Newport",
  "category": "生產力",
  "readDate": "2026-04-03"
}
```

### 補登舊資料規則

若之後要補登以前讀過的書：

- 仍然要照實填寫原始閱讀日期
- 不要把補登日期當成 `readDate`
- 不需要手動調整整份資料檔排序
- build 會自動整理順序

### 作者與類別命名一致性

為了避免搜尋與統計混亂，新增資料時必須保持命名一致：

- 同一位作者使用同一種寫法
- 同一個類別使用同一種名稱
- 不要混用全名、縮寫、不同語言版本，除非專案已明確定義規則

例如：

- `J.K. Rowling` 不要有時寫成 `JK Rowling`
- `自我成長` 不要另一筆寫成 `個人成長`，除非本來就要分成不同類別

### 可接受的原始資料特性

原始資料可以接受：

- 新增順序不完全正確
- 補登舊資料插在任何位置
- 暫時沒有為前端最佳化
- 只保留最基本欄位

build 流程的存在，就是為了把這份人工維護資料整理成適合前端使用的格式。

## 實作原則

### 優先順序

代理實作時，請依以下順序進行：

1. 建立原始資料與產出資料的雙層結構
2. 建立書籍資料整理腳本
3. 串接 build 流程，確保 build 時自動產生前端資料
4. 建立符合深色書店／藝文空間風的基礎視覺樣式
5. 顯示書籍列表
6. 實作書名搜尋
7. 實作作者搜尋
8. 實作日期區間查詢
9. 實作排序
10. 改善 UI 與可用性
11. 補上基本測試或驗證邏輯（若專案已有測試機制）

### 變更原則

- 不要引入後端
- 不要加入不必要的外部服務
- 不要把資料保存改為雲端或資料庫
- 不要為了小功能引入過重依賴
- 優先修改現有結構，而不是無限制擴張檔案數量

### 程式碼風格

- 命名清楚
- 單一職責
- 避免重複邏輯
- 避免過早抽象
- 可讀性優先於炫技寫法

## 篩選與排序邏輯規則

代理在實作時應遵守以下資料流：

1. build 先從 `raw/books.json` 產生 `generated/books.json`
2. 前端從 `generated/books.json` 載入資料
3. 前端先套用單一關鍵字搜尋（同時查書名與作者）
4. 前端再套用日期區間篩選
5. 前端最後套用畫面排序
7. 將最終結果渲染到畫面

不要把搜尋、篩選、排序拆成互相衝突的獨立資料來源。

---

## 容錯要求

- `raw/books.json` 格式錯誤時，build 應顯示合理錯誤訊息
- `generated/books.json` 為空陣列時，頁面不可崩潰
- 日期條件不完整時，前端仍需可正常使用
- 未輸入搜尋條件時，預設顯示全部書籍
- 前端不可假設原始資料一定完美無誤，應依賴 build 後的穩定產物

---

## 效能要求

本專案規模預期不大，優先考慮簡單與可維護性。

但仍應避免：

- 不必要的重複排序
- 不必要的深層 watcher
- 在 template 寫過多複雜邏輯
- 前端每次啟動都自行重做大量資料清理

應盡量把資料整理成本放在 build 階段，而不是前端執行階段。

## 驗收標準

完成後必須符合以下條件：

- 能從 `src/data/raw/books.json` 讀取原始書籍資料
- build 時能自動產生 `src/data/generated/books.json`
- 原始資料順序混亂時，產出資料仍可被正確整理
- 原始資料有格式錯誤時，build 會失敗並提示原因
- 能顯示完整書籍列表
- 能依書名搜尋
- 能依作者搜尋
- 能依閱讀日期做區間查詢
- 能依書名、作者與日期排序
- 搜尋、篩選、排序能同時作用
- 後續新增一本書時，我只需修改 `raw/books.json`
- 前端不直接依賴原始資料檔
- 畫面符合深色書店／藝文空間風
- 不呈現強烈 AI 產品感
- 在靜態部署環境可正常運作
- 沒有後端依賴
